# 事件提取 Prompt

## 任务

你将收到一段聊天记录（按时间分段的消息窗口），以及已有事件的摘要。

你的任务是从这段对话中提取**所有可识别的离散事件**——两个人之间发生的具体事情。

**原则：**
- 只提取有据可查的内容，从对话中能推断出来的场景和细节
- 一个消息窗口可能包含 0 个、1 个或**多个事件**
- 如果消息内容只是日常闲聊（"在吗""吃了吗""晚安"），不构成独立事件，跳过
- 感官细节可以从场景推断（比如提到去了火锅店 → 火锅的气味是合理的推断）
- 情感意义从对话语气和内容推断

**重要：多事件提取**
- 一个窗口中经常包含多个独立事件，你必须**逐个识别并提取**
- 常见的多事件场景：
  - 先讨论了周末计划，后来又吵了一架 → 两个事件（约会计划 + 争吵）
  - 她分享了工作上的事，然后又说了和闺蜜的矛盾 → 两个事件
  - 聊了旅行安排，中间穿插了日常关心，最后约了吃饭 → 旅行计划 + 吃饭约定
- **不要**把整个窗口概括为一个大事件
- **不要**只提取"最重要"的那个事件而忽略其他
- 日常关心（"吃了吗""早点睡"）不构成独立事件，但如果有具体的情感表达（"今天特别想你"）可以作为事件的一部分

---

## 输入格式

你会收到：

1. **消息窗口**：`[{"sender": "her"/"me", "content": "...", "timestamp": "..."}]`
2. **已有事件摘要**（可能为空）：列出已有事件的 id、标题、时间范围、摘要

---

## 输出要求

对每个提取到的事件，输出一个 JSON 对象。**如果窗口中有多个事件，必须输出一个 JSON 数组**（不要只返回第一个事件）。

只有一个事件时：
```json
{ "id": "new_001", ... }
```

多个事件时（必须用数组）：
```json
[
  { "id": "new_001", ... },
  { "id": "new_002", ... }
]
```

```json
{
  "id": "new_001",
  "title": "事件的简短标题（2-6 个字）",
  "time_range": {
    "start": "YYYY-MM-DD",
    "end": "YYYY-MM-DD"
  },
  "timeline": [
    {"time": "YYYY-MM-DD 或 HH:MM", "detail": "这个时间点发生了什么，用'你们'开头描述"},
    ...
  ],
  "what_happened": "用 2-5 句话描述整个事件，用'你们'开头，有画面感",
  "important_details": [
    "一个值得注意的细节",
    "另一个细节",
    ...
  ],
  "sensory": [
    "感官描述（味道/声音/气味/触感/视觉）",
    ...
  ],
  "emotional": "这个事件的情感意义，1-2 句话",
  "source_messages": ["时间戳1..时间戳2"],
  "related_events": ["evt_000"],
  "tags": ["标签1", "标签2"],
  "incomplete_dialogue": false
}
```

---

## 字段说明

### id
- 如果这是一个**全新事件**：用 `"new_001"`, `"new_002"` 等临时 id
- 如果这是**已有事件的续集**：用已有事件的 id（如 `"evt_003"`）

### title
简短有画面感。好的例子：「厦门旅行」「第一次大吵」「圣诞节约会」「她生日那天」
坏的例子：「一次旅行」「一次争吵」「一次约会」

### time_range
事件发生的时间范围。如果只有一天，start 和 end 相同。

### timeline
事件中的关键时间点。每个时间点是一个小场景：
- `time`：具体日期或时间
- `detail`：用"你们"开头描述发生了什么，包含具体细节
- 3-8 个时间点为佳，取决于事件跨度

### what_happened
2-5 句话的整体叙述，有画面感。好的例子：
「你们一起去了厦门，住在曾厝垵附近。三天两夜，逛了鼓浪屿、环岛路、中山路。她带了一个小本子随时画速写，最后一天下雨了，你们在咖啡店坐了一下午。」

### important_details
值得注意的具体细节，这些细节让事件变得独特：
- 她说了某句印象深刻的话
- 某个意外发生的事
- 某个只有你们才注意到的瞬间
- 3-8 个细节为佳

### sensory
从场景中可以推断的感官细节：
- 味道/气味：火锅的辣味、她身上的香水、雨后泥土的气息
- 声音：她笑的声音、海边的浪声、咖啡店的背景音乐
- 视觉：她穿的那件衣服、夕阳的颜色、雪花的样子
- 触感：她手的温度、风吹在脸上、椅子的硬

### emotional
这个事件对关系的意义。不是"她开心"这种泛泛的描述，而是：
「那几天是你们在一起最轻松的时候，她笑得比平时多很多。」

### source_messages
用时间戳范围标记来源消息：`["2022-08-05 08:00:00..2022-08-08 22:00:00"]`

### related_events
关联的已有事件 id。一个事件可以关联多个已有事件：
- 如果这是某个事件的续集：`["evt_003"]`
- 如果这同时是两个事件的续集（比如吵架的和好 + 旅行计划的实现）：`["evt_003", "evt_005"]`
- 如果是全新事件且无关联：`[]`

### tags
从以下标签中选择（也可以自定义）：
- `旅行` `约会` `吵架` `和好` `节日` `纪念日` `日常` `惊喜` `感动`
- `高峰期` `稳定期` `磨合期` `疏远期` `转折点`
- `第一次` `最后一次`

### incomplete_dialogue
布尔值。如果消息窗口中只有一方的消息（对话上下文不完整），设为 `true`。默认 `false`。
详见下方「单方面消息处理策略」。

---

## 续集检测规则

对比已有事件摘要，判断当前事件是否为某个已有事件的续集：

1. **话题延续**：如果对话中提到之前讨论过的事（"上次说的那个旅行"→ 是之前旅行讨论的续集）
2. **情绪延续**：如果这是之前吵架/冷战的后续（和好、道歉）
3. **计划执行**：如果这是之前约定的计划的实现（之前约好去旅行 → 现在真的去了）

判断为续集时：
- `id` 使用已有事件的 id
- `related_events` 包含被续的事件
- `timeline` 只包含新的时间点（不重复已有事件的内容）

---

## 无事件时的输出

如果消息窗口中没有可提取的事件（只是日常闲聊），输出：

```json
{"skip": true, "reason": "日常闲聊，无具体事件"}
```

---

## 单方面消息处理策略

如果消息窗口中只有 `her` 的消息（没有 `me` 的回复），说明对话上下文不完整。

**处理规则：**

1. **仍然可以提取事件**：单方面消息也能反映发生了什么（她抱怨了什么、她分享了什么、她的情绪变化）
2. **在事件 JSON 中标注**：添加 `"incomplete_dialogue": true` 字段
3. **what_happened 写法调整**：用"根据她的消息推断"开头，而非"你们"
   - 好的写法：「根据她的消息推断，那天她心情不好，可能是因为你回消息太慢了」
   - 不好的写法：「你回消息太慢了，她很生气」（你并不知道你做了什么）
4. **timeline 只包含她的时间点**：因为没有你的回复，timeline 只能记录她发消息的时间
5. **important_details 区分原文和推断**：
   - 原文：直接引用她说的话
   - 推断：用「（推断）」标注，如「（推断）她可能在等你主动联系」

---

## 注意事项

- 温柔对待感情内容，不评价对错
- 引用她原话时保持原样
- 不确定的细节用"好像""记得是"标注
- 如果消息只有单方面（只有 her 或只有 me），也可以提取事件，但标注"对话不完整"
